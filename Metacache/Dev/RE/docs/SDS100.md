## TL;DR for the multi-scanner backend

1. **Folder structure is identical** to the BT885's BCDx36HP card on
   disk - same `BCDx36HP/` parent, same `firmware/CityTable_...dat`
   and `firmware/ZipTable_...dat`, same per-state `s_*.hpd` shape,
   same `hpdb.cfg` state index.
2. **`TargetModel` does NOT identify the scanner model** - it
   identifies the **firmware family** (`BCDx36HP`). Both BT885 and
   SDS100 write `TargetModel\tBCDx36HP` in their HPD/CFG headers.
   (The codebase currently assumes BT885 writes
   `TargetModel\tBeartracker885`. This is at minimum incomplete and
   may be flat wrong for real BT885 cards. See
   "Identity contradiction" below.)
3. **The reliable model fingerprint is `scanner.inf`'s `Scanner`
   record** - field 1 is the literal model string (`SDS100` here).
   Secondary: `profile.cfg`'s `ProductName` row.
4. **HPD record types are a strict superset** of what the BT885 uses.
   The current parser in `scanner_manager.py:411-545` already handles
   every record we observed in conventional HPDs. Trunked HPDs use
   `Trunk`, `Site`, `T-Freq`, `T-Group`, `TGID` - all already parsed.
5. **What's NEW on SDS100** = the *content* the BT885 lacks UI for:
   - the giant `profile.cfg` (BT885 has no equivalent)
   - `app_data.cfg` last-state (BT885 doesn't write last-state)
   - `discvery.cfg` (sic) discovery config stub
   - populated `f_list.cfg` + `favorites_lists/f_*.hpd`
     (BT885 ships an empty 42-byte `f_list.cfg` stub but no payloads)
   - 36-slot `ServiceType` mask (BT885 maps to 14 service-type IDs and only plays 5)

   **The empty directory skeleton is NOT new on SDS100.** Both BT885
   and SDS100 carry `activity_log/`, `alert/`, `audio/`,
   `discovery/`, `favorites_lists/` empty-stub directories. They are
   part of the BCDx36HP firmware spec; the BT885 just never
   populates them. (See `docs/SD_CARD_COMPARISON.md`.)
6. **HPDB layout is the same.** Both the per-state `s_*.hpd` files
   and `hpdb.cfg` live under `BCDx36HP/HPDB/` on **both** scanners.
   `Bt885Profile.card_identity_files` should be updated to
   `["BCDx36HP/HPDB/hpdb.cfg", "BCDx36HP/scanner.inf"]` (currently
   just `["hpdb.cfg"]`).

### Identity contradiction (RESOLVED 2026-04-27)

- **Codebase says** (in `Bt885Profile.target_model_aliases`):
  `("Beartracker885", "BearTracker885", "BT885")`. Tests in
  `tests/test_sdcard.py`, `tests/test_metastore.py`, and
  `tests/test_merge_and_zip.py` all use
  `TargetModel\tBeartracker885` as the BT885 fixture.
- **The real BT885 firmware writes** `TargetModel\tBCDx36HP` in
  every header (verified 2026-04-27 against a real BT885 SD card on
  `<HOST>`; see [`docs/BT885.md`](BT885.md)).
- **The codebase aliases are stale.** Possibility #1 from the prior
  list is what real hardware does. Possibility #2 / #3 are dead.
- **`TargetModel` alone is insufficient** to distinguish models on a
  multi-scanner card. The driver layer must read `scanner.inf`
  `Scanner` field 1 (`BT885-SCN` vs `SDS100`) or - as a fallback -
  `profile.cfg`'s `ProductName` row (only present on SDS100).
- Test fixtures using `TargetModel\tBeartracker885` need to be
  updated to `TargetModel\tBCDx36HP` once the model-detection
  refactor lands; they are wrong about real hardware behavior.

## `BCDx36HP\favorites_lists\` - SDS100 Favorites system

The SDS100 (and SDS200, BCD436HP, BCD536HP, etc.) supports the
**Favorites Lists** feature - up to ~256 user lists with quick-key
toggling. The BT885 cannot use this feature even though the file
format is the same on disk; it has no hardware buttons or UI for
favorites, so it just doesn't read this folder.

### `favorites_lists\f_list.cfg` - master manifest

```
TargetModel	BCDx36HP
FormatVersion	1.00
F-List	<COUNTY>	f_000001.hpd	Off	Off	...   (115 more On/Off slots)
F-List	Home	f_000002.hpd	Off	On	...
F-List	Williston	f_000003.hpd	Off	Off	...
```

Each `F-List` record:

| Idx | Value (sample) | Meaning |
| --- | --- | --- |
| 0 | `F-List` | record type |
| 1 | `Home` | display name |
| 2 | `f_000002.hpd` | file under `favorites_lists/` containing the list payload |
| 3 - 118 | `Off`/`On` (116 slots) | quick-key enable mask, per scan position. Index 0 unused, indices 1-115 are Q-Key 1 through Q-Key 115 (or similar - exact mapping TBD; lots of trailing tab whitespace). |

For Home (line 4 in the dump), index 1 is `On` and the rest are
`Off` - matches the SDS100's "list assigned to Quick Key 1" UX.

### `favorites_lists\f_NNNNNN.hpd` - per-list payload

Same record format as `s_*.hpd` BUT the records inside are
favorites-specific. Sample `f_000002.hpd` (Home):

```
TargetModel	BCDx36HP
FormatVersion	1.00
Trunk			<TRUNK_SYSTEM>	Off		P25Standard	Off	Off	Auto	Ignore	Srch	Off	Off	0	Off	Off	Analog	Off	Off	On	NEXEDGE
DQKs_Status		On	On	On	On	... (100 slots)
Site			Simulcast	Off	<LAT>	<LON>	24.0	AUTO	Standard	Wide	Circle	Off	400	Auto	8	Off	4D2	Global
T-Freq				Off	851187500	1	Srch
T-Freq				Off	851775000	2	Srch
... (more T-Freqs - control channels)
T-Group			<AGENCY>	Off	<LAT>	<LON>	24.0	Circle	Off
TGID			A1 Primary	Off	2147	ALL	2	2	0	Off	Auto	Off	On	Off	Off	Any
TGID			A2 Secondary	Off	2149	ALL	23	2	0	Off	Auto	Off	On	Off	Off	Any
... (more TGIDs)
```

Differences from a state HPD:

- **No `AreaState` / `AreaCounty` rows.** Favorites are user-curated;
  there's no county/state grounding.
- **No `Id` fields on records.** A state HPD has things like
  `CGroupId=24005`, `CFreqId=89622` - autogenerated by the database.
  Favorites HPDs have `T-Group\t\t\t<name>...` (two empty fields
  where IDs would go) and `TGID\t\t\t<name>...` likewise. The scanner
  manages favorites by name, not by RR ID.
- **`DQKs_Status` row** appears immediately after each `Trunk` row.
  100 On/Off slots = "Department Quick Keys" - per-talkgroup-group
  hotkey enables. The state HPDs do not have this row.
- **Trunk record fields** carry extra columns vs the typical state
  HPD trunked entry (e.g. `NEXEDGE` token at the end indicates
  multi-protocol awareness). Document fields fully when we start
  editing favorites.

### Implication for the driver layer

The BT885 profile should advertise `supports_favorites = False` (or
similar) and refuse to enumerate this folder. The SDS100 profile
should claim it. The on-disk records are the same shape as the
existing parser - we can lift the parser as-is, the only new
behavior is "this is a favorites HPD, not a state HPD" which the
caller already knows from the path.

## `BCDx36HP\firmware\` - firmware tables

```
firmware/
  CityTable_V1_00_00.dat    566,492 B
  ZipTable_V1_00_00.dat     693,758 B
```

Identical names and approximate sizes to the BT885's firmware
folder. `sdcard.py:_content_fingerprint` already digests the first
1 MB of these files plus the detected TargetModel; that hash will
collide between BT885 and SDS100 cards if the firmware ZipTable /
CityTable versions match (which they often do, because both
scanners are the same chipset and ship the same DB build).

This means the **content fingerprint is not a model differentiator**
for BCDx36HP-family scanners. Volume serial + scanner.inf is the
real fingerprint.

## `BCDx36HP\app_data.cfg` - last-active scan state

```
TargetModel	BCDx36HP
FormatVersion	1.00
ModeInfo	IDscan
ScanListType	FavoritesList	Home
ScanTrunkSystem	Off	Trunk			<TRUNK_SYSTEM>	Off		P25Standard	Off	Off	Auto	Ignore		Off	Off	0	Off	Off	Ignore	Off	Off	On	IDAS
ScanT-Group	Off	T-Group			<AGENCY>	Off	<LAT>	<LON>	24.0	Circle	Off
ScanSite	Off	Site			Simulcast	Off	<LAT>	<LON>	24.0	AUTO			Circle	Off	400	Auto	8	Off	4D2	Global
ScanT-Freq	T-Freq				Off	853187500	0	Srch
```

This file mirrors the scanner's last-active state when it was
powered down: which mode (`ModeInfo IDscan` = ID Scan), which list
(`Home`), which trunk system / talkgroup-group / site / freq was
in focus. The BT885 doesn't write a comparable file - it's just
"on or off" at power.

Treat this as ephemeral state. Don't merge or version it; on a
push to the card, copy verbatim.

## `BCDx36HP\profile.cfg` - the giant SDS100 settings file

15,612 bytes, 184 lines. This is where every SDS100-specific
feature lives. The full file is worth reading; the headlines:

| Record | Slots | Purpose |
| --- | ---: | --- |
| `ProductName` | - | Model name (`SDS100`). Best secondary fingerprint after `scanner.inf`. |
| `GlobalSetting` | 20 | Global toggles (key beep, opening msg, charge time, contrast, attenuator, etc.). Field shapes TBD. |
| `SearchCommon` | 12 | Search-mode global params (modulation, attenuator, NAC, repeater find, etc.). |
| `PresetBroadcastScreen` | 5 | Pre-defined broadcast bands to skip (FM, weather, ...). |
| `CustomBroadcastScreen` | 30 | 10 user-defined skip-bands (low/high triplets). |
| `CurrentLocation` | 3 | Last GPS fix the unit had (lat, lon, accuracy radius). |
| `GpsOption` | 2 | GPS coordinate display (`DMS`/`DEG`) + serial baud (`4800`). |
| `ServiceType` | **36** | Per-service-type On/Off mask. **SDS100 supports 36 service types vs BT885's 14.** |
| `CustomServiceType` | 10 | 10 user-defined service-type slots. SDS100-only. |
| `Weather` | 4 | Weather-priority + alert mode + SAME mode. |
| `WxSameList` | 5 rows x 9 codes | SAME county codes for weather alerts. |
| `DisplayOption` | many | Display layout choice (DEC/HEX freq, AFS/decimal IDs, color theme). |
| `Backlight`, `Battery`, `OwnerInfo`, `ClockOption` | - | Self-explanatory hardware settings. |
| `RecordingOption` | 2 | Record duration / replay length. |
| `StandbyOption` | 2 | Standby mode + power-on state. |
| `LimitSearch` | 10 rows x ~16 fields | 10 user-defined limit-search ranges (low/high freq, mod, step, attenuator, etc.). |
| `CloseCall` | ~16 | Close Call (RF nearest-frequency capture) settings. |
| `ToneOut` | 32 rows | 32 paging tone-out slots (freq, decode tones, mod). |
| `Waterfall`, `CustomWfBand` x 10 | - | SDS100 waterfall display. **Not on BT885.** |
| `WfColors` | 6 | Waterfall color palette (hex). |
| `IfxFreqs` | - | IFX exchange/intermod frequencies. |
| `BandDefault` | 31 rows | Per-band default modulation + step. |
| `DispOptItems` | many rows | Configurable per-screen display layouts (1-7 = different scan screens). Each row tells the firmware which fields to show in which slot. |
| `DispColors` | many rows | Per-screen color overrides. |
| `QuickKeys` | 100 | Quick-key enable mask. **Not on BT885.** |

Fields per record vary; we have not exhaustively decoded each.
Capture the verbatim row when round-tripping, **do not** rebuild
from a minimal field set, or the scanner will reject the file or
silently revert to defaults.

### Sample - the 36-slot `ServiceType` row

```
ServiceType	On	On	On	On	Off	On	On	On	On	Off	On	On	On	On	On	On	On	Off	Off	On	On	On	On	On	On	On	Off	Off	On	On	On	On	On	On	Off	Off	On
```

Maps to service-type IDs 1..36 (or possibly 0..35 - need to verify
against the Sentinel app). The BT885 profile only enumerates IDs
1, 2, 3, 4, 6, 7, 8, 9, 14, 22, 23, 24, 25, 209 - a sparse subset.
Either:

- The numbering on disk is dense (1..36) and the BT885's "209" in
  the codebase is a UI-only label, not a wire ID. Need to verify.
- Or the SDS100 mask is positional (slot N in the file = SDS100
  service-type ID N) and the BT885's 209 is genuinely a
  scanner-specific extension. Seems less likely.

Research priority before adding the SDS100 profile.

```python
        if parts and parts[0] == "Scanner" and len(parts) > 1:

    for line in path.read_text(encoding="utf-8", errors="replace").splitlines():
```

clean signal we can use in the desktop detector: presence of CDC ports
with VID `0x1965` plus PIDs `0x0019`/`0x001A`, **without** a Uniden
mass-storage volume = scanner is in serial mode of some flavor.)

## Findings: command surface in this mode

Out of 36 whitelisted query commands, only **3 elicit a real response**.
Everything else times out at 600 ms with no bytes returned.

| Sent | Response (verbatim) | Notes |
|---|---|---|
| `MDL\r` | `SDS100-SUB\r` (×4) | model identifier, repeated 4 times |
| `VER\r` | `Version 1.03.05 \r` (×4) | firmware version, trailing space |
| `MEM\r` | `SDS100-SUB\r` (×4) | **stale buffer** - returned the cached MDL response |
| `VOL\r` | `Version 1.03.05 \r` (×4) | **stale buffer** - returned the cached VER response |
| `MDL,?\r` | `SDS100-SUB\r` (×6) | same payload, 6× repetition |
| `VER,?\r` | `Version 1.03.05 \r` (×6) | same payload, 6× repetition |
| `VOL,?\r` | `Version 1.03.05 \r` (×6) | stale buffer again |
| all others | (no response) | `STS`, `GLG`, `PWR`, `SQL`, `BAV`, `BLT`, `CNT`, `DMA`, `SCN`, `CBP`, `CSP`, `LOC`, `GIN,GPS`, `CLK`, `OMS`, `BLI`, `RST`, `PRI`, `ALT`, `GID`, `NTG`, `WFL`, `FAV`, `GST`, `RLG`, `STS,?`, `GLG,?`, `SQL,?`, `RSI` |

## Three observations that say "this is NOT normal Remote Mode"

1. **Model string ends in `-SUB`.** The SDS100 has a dual-processor
   architecture - **MAIN** (UI, SD card, scan engine) and **SUB**
   (radio front-end / DSP). `SDS100-SUB` is the SUB processor identifying
   itself.
2. **No command-name prefix on responses.** Standard Uniden Remote
   Protocol replies are `MDL,SDS100\r` (the request mnemonic is echoed
   back). Here we get just `SDS100-SUB\r` - a different framing.
3. **Repeated payloads + buffer leakage.** Successful responses arrive
   4 times (later 6 times); commands without native handling return the
   *previous* command's payload. That is consistent with a bootloader
   / firmware-updater echo buffer, not a parse-and-respond command shell.

**Conclusion (high confidence):** The scanner is in **SUB-processor /
firmware-updater mode**, not the normal "Remote (Serial)" control mode.
PID `0x0019` likely identifies this mode specifically; the normal Remote
Mode is expected to enumerate with a different PID.

## What this means for the project

- **Updater mode is NOT what we want for command-surface RE.** It only
  supports identity + flash routines, which is why 33 of 36 commands
  silent-failed. We will not learn how the scanner *runs* from this
  mode; only how it *gets reflashed*.
- **There is no harm done.** The probe was strictly read-only. Even
  in updater mode, sending `MDL`/`VER`/`MEM` is safe - those are
  identity reads.
- **Next step is to enter normal Remote (Serial) Mode**, where the
  full Uniden Remote Command Protocol is exposed:
  `MDL,SDS100<CR>` reply, plus `STS`, `GLG`, `PWR`, `KEY` (avoid),
  `GLT` (avoid - mutates), `EPG`/`PRG` (avoid - changes state),
  `RMB`, `WIN`, `MNU`, `JPM`, `JNT`, etc.

## How to enter Remote (Serial) Mode on the SDS100

When the scanner powers on with a USB cable connected, it shows a
connection-mode prompt on the LCD. On firmware 1.23.07 the keys are:

- **Mass Storage** - exposes the SD card as drive (`D:\`).
- **Serial** (period key on the keypad) - enumerates **two** CDC ACM
  ports simultaneously (see below).

There is no separate "Update" option on the boot prompt; SUB-processor
flashing happens through the SUB endpoint that's enumerated alongside
the MAIN endpoint in Serial mode.

> Correction (later lab): PID `0x0019` is the SUB command port (not a bootloader); Sentinel firmware updates are UMS/SD-card only — see Session 4.

## CRITICAL DISCOVERY: Two CDC ports, two different protocols

When the SDS100 is in Serial mode, **two virtual COM ports appear**,
both with Uniden VID `0x1965`, but with **different PIDs and very
different protocols**:

| PID | Role | Protocol |
|---|---|---|
| `0x0019` | **SUB processor bootloader** | Per-byte echo, recognizes only first character: `M*` -> model, `V*` -> version. Used by Sentinel for SUB firmware flashing. NOT a general-purpose protocol. |
| `0x001A` | **MAIN processor command port** | **Full Uniden Remote Command Protocol** with proper `cmd,result\r` framing. This is where everything lives. |

On Windows, both enumerate as plain "USB Serial Device (COMn)" with no
distinguishing description text, so PID is the only reliable way to
tell them apart. Use:

```powershell
Get-CimInstance Win32_PnPEntity |
  Where-Object { $_.DeviceID -match 'VID_1965' } |
  Select-Object Name, DeviceID
```

We initially assumed `0x001A` was a GPS NMEA stream (older Uniden
scanners use a separate GPS CDC port). It is not - it stays silent
until queried.

```
                        ,

______________________________,
                        ,

______________________________,

                        ,
                        ,
                        ,

                              ,
```

Working hypotheses (need more data to confirm):

- The first field after `STS,` is a **status-flags bitfield** (14 bits
  here). Best guesses for bit positions: HOLD, ATT, P25, GPS, Battery,
  Lock, Priority, Squelch-met, Recording, etc. To map: toggle each
  feature one at a time and diff.
- Bytes `0xAC`/`0xAD` and `0x1A`/`0x1B` are scanner-private icon
  glyphs in the Uniden custom code page (BAR / arrow indicators).
- The **fixed 30-character lines of `_` and ` `** are the LCD's
  horizontal rules and blank rows - useful for layout but not data.
- The **trailing numeric tail** (`1,1,0,0,,,5,OFF,3`) looks like
  selected-menu indices and possibly contrast/backlight values.
  `OFF` is probably a recording flag.

## PWR payload format

```
PWR,-76,08531875
```

- `-76` = received-signal-strength in dBm (signed integer).
- `08531875` = frequency. Almost certainly **decimal Hz / 100** (i.e.
  853.1875 MHz, expressed as `f * 1e4`). Compare with known
  trunked-system frequencies on the <TRUNK_SYSTEM> system to confirm.
  **Plausible**: 853.1875 MHz is in band 14 (851-869 MHz) which is
  exactly where Florida's P25 trunked simulcast NACs (incl 0x4D2 / 1234
  decimal) live. Match.

## GLG payload format

```
GLG,,,,,,,,,,,,
```

12 empty comma-separated fields. When the scanner is *actively
receiving* a transmission, this returns populated values. Documented
Uniden BCDx36HP `GLG` schema (best known):

```
GLG,FRQ_TGID,MOD,ATT,CTCSS_DCS,NAME1,NAME2,NAME3,SQL,MUT,SYS_TAG,CHAN_TAG,P25_NAC
```

Need a transmission in progress to capture a real example. The user
should hold the scanner near an active system (railroad, public
safety) and re-probe.

```
MEM  RST  PRI  ALT  GID  NTG  WFL  FAV  RLG
```

## Commands where `,?` form is interesting


read-only by nature. This is a safe inference -- we are NOT going to
test it by sending an actual write.

# Next-step plan: expand the read-only command surface

The 8-command result is a beachhead. The Uniden BCDx36HP / SDS100
Remote Command Reference (publicly documented by Uniden in
"Operation Specification" PDFs floating around the radio-hobbyist
community, eg. RadioReference wiki) lists many more *passive* queries.
Candidates to add to the whitelist for the next pass, **after vetting
each one against documentation as read-only**:

- `RMB` - remote-mode banner (some FW returns model + state).
- `WIN` - LCD window contents (different layout from STS).
- `DGR` - get GPS location (returned by scanner if GPS is connected).
- `RIN` - radio info banner (model + serial + FW + flash version).
- `BTV` - battery voltage (some FW use BTV not BAV).
- `LCB` - LCD backlight state (read).
- `BSV` - battery save mode (read).
- `MNU` - menu structure (highly experimental; on some FW it dumps
  the menu tree, on others it walks it - need careful test).
- `EWP` - error / warning probe (some FW use this).
- `PSI` - profile system info.
- `SUM` - storage usage (memory summary).

Strict rules for the next pass:

1. **Stay on COM6 (PID `0x001A`)**. COM5 is the SUB bootloader; we

3. **`PRG` stays forbidden** until we explicitly decide we want to
   inspect Programming Mode; even though `EPG` exits cleanly, entering
   `PRG` halts scanning and is a state change.
4. **Capture `STS`/`GLG` snapshots while different things happen** on
   the scanner (active reception, hold, lockout, menu open). Diff the
   payloads to map the status-bit layout.

When the bigger surface map is in hand, fold it into

`Metacache/Dev/MULTI_SCANNER_BACKEND.md` as the eventual basis for an
optional "live scanner" feature in the multi-scanner backend

### `,?` form semantics (refined)

Sending `cmd,?` on FW 1.23.07 yields exactly two outcomes:

- `cmd,OK` -> command has a write form. Observed for `VOL,?` and
  `SQL,?` (both confirmed Get/Set in V2.00 spec).
- `cmd,ERR` -> command has no write form. Observed for the spec's
  read-only ones (`MDL`, `VER`, `STS`, `GLG`) AND, surprisingly, for
  spec-documented Get/Set commands (`SVC,?`, `DTM,?`, `LCR,?`,
  `FQK,?`).

That last bullet is the interesting one: the V1.02 spec lists
`SVC`/`DTM`/`LCR`/`FQK` as Get/Set, yet `,?` on them returns ERR.
Interpretation: the `,?` write-acceptance probe is a vestigial
BCDx36HP-era syntax that only the legacy commands (VOL/SQL) honor;
the modern SDS commands accept their write-form directly without
the `,?` handshake. This is consistent with `,?` being a non-mutating
probe and **does not change our safety stance** - we still never
escalate from `,?` to a real write.

## GLT pagination

GLT responses larger than ~11 records arrive as **multiple back-to-back
XML chunks** in a single response burst. Each chunk is a complete XML
document terminated by `<Footer No="N" EOT="0|1"/>`. `EOT="0"` means
"more pages coming"; `EOT="1"` means "this is the last page". The
scanner streams all pages automatically - we don't need to send a
"next page" request. Observed during Session 3:

- `GLT,FTO` -> 32 records over **3 pages** (No=1 EOT=0, No=2 EOT=0, No=3 EOT=1)
- `GLT,CS_BANK` -> 10 records over **2 pages** (No=1 EOT=0, No=2 EOT=1)

A parser that wants the full list MUST concatenate `<FTO>` (or
`<CS_BANK>`) elements across all chunks, and stop when it sees
`EOT="1"`.

## New commands - V2.00

### `KAL` Keep Alive (read-only / no response)

```
Controller -> Radio:  KAL\r
Radio:                (no response)
```

Pure heartbeat. The spec explicitly says "Radio does not return a
response for Keep Alive command." Useful when the controller wants
to keep the scanner from timing out the serial session during a
long-idle window. Add to QUERIES with a "no response expected" hint.

### `POF` Power OFF (mutating - already on FORBIDDEN list)

```
Controller -> Device: POF\r
Device:               POF,OK\r
```

Already on `FORBIDDEN_FOR_READ_ONLY`. Noted here for completeness;
**never send.**

### `GCS` Get Charge Status (read-only - high value!)

```
Controller -> Device: GCS\r
Device:               GCS,CST=[CST],VOLT=[mV]mV:[%]%,CURR=[mA]mA,TEMP=[C]C\n
```

Note the response terminator is `\n` (LF) not `\r` (CR) - **the only
command in the entire spec that uses LF**. The probe's read loop
already handles both terminators, but parsers that assume CR-only
will break. Documented for the AI brain.

`[CST]` charge sub-state values (per spec):

| CST | Meaning |
|---|---|
| 0 | `CHG_SUBSTS_NO_CHG` - no external power or no battery |
| 1 | `CHG_SUBSTS_BQ274_INIT` - initializing the BQ27426 fuel gauge IC |
| 2 | `CHG_SUBSTS_TEMP_NG` - abnormal temperature |
| 3 | `CHG_SUBSTS_PWR_NG` - abnormal power |
| 4 | `CHG_SUBSTS_FULL_CHG` - full charge |
| 5 | `CHG_SUBSTS_RECHG` - recharge |
| 6 | `CHG_SUBSTS_CHARGE` - charging |

`VOLT` is millivolts, `:` then percent remaining. `CURR` is signed
milliamps (positive = charging, negative = discharging). `TEMP` is
Celsius. **This single command replaces every speculative `BAV`,
`BLT`, `BTV`, `BSV` we tried in Sessions 1-3.**

### `GW2` Get Waterfall FFT without separator (mutating - stream toggle)

```
Controller -> Radio: GW2,[TYPE],[ON/OFF]\r
Radio -> Controller: GW2,[FFT_DATA]\r  (binary 240-byte FFT frames, while ON)
```

Same shape as `GWF` but the response carries no per-frame separator -
the receiver has to know the frame size from the stream. Same stream
toggle semantic - turning it ON leaves the scanner pushing frames
until you turn it OFF. **Forbid for passive RE** (already covered by
the existing `GWF` forbid; add `GW2` to the explicit list when we
upgrade the probe for V2.00).

## Bombshell: PID 0x0019 is the SUB processor command port, NOT a bootloader

The **Session 1 conclusion that PID `0x0019` was firmware-updater /
bootloader mode is wrong.** Sessions 1-3 left this as an open
question; Session 4 resolved it conclusively. The corrected facts:

1. **Firmware updates do NOT use any USB protocol.** They are
   delivered by SD-card file-drop (see "Firmware update mechanism"
   below). There is no proprietary USB bootloader handshake to RE.
2. **PID `0x0019` is a permanent USB CDC endpoint** that the scanner
   exposes whenever it's in serial mode. It is a sister port to
   `0x001A`, but routed to the **SUB MCU** (NXP LPC43xx) instead of
   the MAIN MCU. Confirmed via:
   - `MDL` -> `SDS100-SUB\r` (different model string from the MAIN
     port's `SDS100\r`)
   - `VER` -> `Version 1.03.05\r` initially, `Version 1.03.15\r`
     after the sub firmware update (it's the SUB version)
   - The 6-of-36-commands silent-failure pattern that earlier looked
     like "bootloader limited" was actually the SUB port having a
     much smaller command set than MAIN. SUB only knows identity +
     status + a few RF/DSP commands.
3. **Firmware quirk: SUB port responses arrive 4x echoed.** Sending
   `MDL\r` produces `SDS100-SUB\rSDS100-SUB\rSDS100-SUB\rSDS100-SUB\r`.
   Almost certainly an unflushed FIFO on the SUB side. Parsers must
   take only the first line. (Earlier sessions said "later 6 times"
   - we suspect that was a mode-dependent buffer state.)
4. **Most MAIN-protocol commands time out on the SUB port.** Sending
   `STS`, `GSI`, `GLG`, `PWR`, `SQL` to COM3 returns nothing within
   1.5 s. SUB answers a different (smaller) Remote-Command surface.
5. **V2.00 spec commands `GCS` and `KAL` returned `ERR` on MAIN
   (COM4)** even on FW 1.26.01. Hypothesis: they're SUB-targeted
   (charge status is a power-management concern, owned by the SUB
   processor) - but they also returned no response on COM3. Either
   the spec is forward-looking for a future SDS150 / SDS200 build,
   or the commands need a specific arming context we haven't found.
   **Status: unconfirmed; both errored on FW 1.26.01.**

## Firmware update mechanism (from Uniden's own readme)

This was opaque through Sessions 1-3 because we never opened the ZIP
readmes. Session 4 cracked it open. Procedure:

1. Scanner in **USB Mass Storage** mode (NOT serial mode).
2. Drop the new `.bin` (Main MCU) into `BCDx36HP\firmware\`. NEVER
   touch `CityTable_*.dat` or `ZipTable_*.dat` (those are the dat
   files Uniden's readme warns will brick startup if removed).
3. NEVER place more than one `.bin` in the folder simultaneously
   ("the upload will not start or will not be uploaded correctly").
4. Reboot the scanner. The onboard bootloader on MAIN MCU reads the
   `.bin` from SD on next boot and self-flashes.
5. Scanner deletes the `.bin` after successful flash.
6. Same procedure for Sub firmware with `.firm` extension.

**Sentinel is just a UI wrapper around this procedure.** It puts the
scanner in Mass Storage, copies the file, and prompts the user to
reboot. Nothing magic.

This unblocks an in-app firmware updater - see
`Metacache/Dev/FIRMWARE_UPDATER.md`.

### Phase 1 - SUB-port systematic probe

New script: [`tools/probes/sub_probe.py`](../tools/probes/sub_probe.py). Alphabet-attack with
4x-echo decoding and SUB-specific forbidden mnemonic set. Probes
single letters A-Z, two-letter combos AA-ZZ, ~50 targeted three-letter
combos seeded from the Sub firmware string table.

Sessions stored under `../sessions/sub_probe_*`. Findings:

- **`U`** is the lone genuine undocumented hit on SUB. Returns a
  9-byte response: `U5C42\\r<\\x00><2 binary bytes>`. The 5-byte ASCII
  prefix `U5C42` is stable; the 2 binary bytes change between calls
  (samples: `<\\xC1>P`, `<\\xC0><\\xBC>`, `AC`, `D<\\x92>`, `<\\xC1>,`).

  firmware echoes its identity string (`SDS100-SUB\\r` or
  `Version 1.03.15 \\r`) for any input starting with `M` or `V`,
  with the echo count varying by input length. So `MA-MZ`, `VA-VZ`,
  `MIXG`, `VGA`, `VGM`, `VGN`, `VGAM` all looked like hits but are
  the same partial-match path.
- The Sub firmware contains **35 untriggered printf format strings**
  (status `S%02X%04X...`, gain readouts, FFT/CIC/FIR debug, R840
  modes, register dumps). These are hypothesis targets for future
  probing - the existence of a format string proves there's code
  somewhere that emits it.

  all FLs - undocumented but works on FW 1.26.01! This is a **new
  read-only command** to add to the application.
- 12 other GLT subforms (CC, WX, SWS_FREQ, CCHIT, CS_FREQ,
  QS_FREQ, RPTR_FREQ, UREC_FOLDER, BAND_SCOPE, STGID, DEPT, SITE)
  return empty `<GLT><Footer/></GLT>` - recognized as valid
  keywords but no data in the current scanner state.
- **`GLT,UREC_FILE,0`** TIMES OUT (1505 ms no response). Suspicious
  - possibly a firmware bug. Marked NORESP and flagged "do not use"
  in the catalog.
- **`BFH`** (Band Scope Frequency Hold) is mutating per V1.05 - added
  to `FORBIDDEN_FOR_READ_ONLY`.

### Phase 5 - Cross-correlation

New script: [`_correlate_responses.py`](_correlate_responses.py).
Joins SUB-probe hits against the Sub firmware string table by
converting printf format strings into regexes and matching
responses against them. Output: [`sub_command_response_map.md`](sub_command_response_map.md).

Loaded 38 unique SUB-probe hits, 269 firmware strings, 35 format
strings. The lone `U` hit response (`U5C42\\r<\\x00>...`) didn't
match any firmware format string by regex - which suggests `U`'s
response is emitted by a non-printf code path (likely a fixed-byte
register dump rather than a `printf("U%05X\\r...", val)` call).
Identity-fallback hits (M*/V*) likewise didn't match format strings
since they're hard-coded identity emits.

### Phase 6 - Sub firmware static analysis

Phase 6.1 (extract payload): COMPLETE.
[`tools/firmware/inflate_sub.py`](../tools/firmware/inflate_sub.py) parses the `.firm` container
header, extracts 90,076 bytes of ARM Cortex-M payload from offset
0x80 to 0x16080. **Critical correction to prior assumption**:
the firmware is NOT zlib- or LZMA-compressed; the "zlib magic"
hits in the structure report were coincidental matches in plaintext
ARM machine code. Output:
[`firmware/sub_1.03.15_inflated.bin`](../firmware/sub_1.03.15_inflated.bin),
[`firmware/sub_1.03.15_chunk_map.md`](../firmware/sub_1.03.15_chunk_map.md).

Decoded ARM Cortex-M reset vector:
- **Initial SP**: `0x10020000` (LPC43xx local SRAM bank 0 top)
- **Reset PC**: `0x140001d4` (Thumb-mode entry in SPIFI flash region)
- **All exception handlers** in `0x14000000+` flash range, except
  SysTick at `0x10006295` (in SRAM - copy-on-boot routine).

Phase 6.2 (Ghidra import): NOT STARTED. Full runbook in
[`ghidra_import_runbook.md`](ghidra_import_runbook.md). Per the
plan, this is 20-50 hours of focused work.

Phases 6.3-6.5 (dispatch / inter-MCU bus / handler docs): PARTIAL
via [`_sub_static_analysis.py`](_sub_static_analysis.py). Best-effort
findings without Ghidra:
- 45 ASCII command-mnemonic candidates, 31 clusters - mostly false
  positives (random ASCII in machine code). No obvious dispatch
  table found by clustering alone.
- LPC43xx peripheral 32-bit constant scan:
  - `UART1`: 6 occurrences
  - `SSP0` (SPI): 6 occurrences
  - `ADC1`: 8 occurrences
  - `SCT`: 9 (timer)
  - `USB0`: 5
  - `I2S0/I2S1`: not detected via constant scan (likely accessed
    via struct pointer indirection, not direct literals)
- **Most likely inter-MCU bus**: `UART1` or `SSP0`. Both have ~6
  references, consistent with a control channel that initializes
  once and then runs in interrupt/DMA mode without re-loading the
  base address.
