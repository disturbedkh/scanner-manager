# Uniden BearTracker 885 - SD Card RE

> **Canonical narrative is in the wiki**:
> [`wiki/RE-SD-Card.md`](../../../wiki/RE-SD-Card.md). This file is
> the lab notebook for the BT885 specifically.

> Captured 2026-04-27 on `<HOST>` from a real BT885 SD card
> mounted at `E:\`. Firmware MAIN `1.01.02`, hardware rev `01`,
> serial `<SERIAL>`. All field values quoted verbatim;
> tabs are literal `\t`.
>
> See also: `RE/SD_CARD_COMPARISON.md` for the BT885 vs SDS100 diff,
> and `RE/SDS100.md` for the family superset.

The BT885 was the first scanner this project supported. Most of the
RE listed here is already reflected in `scanner_manager.py` and
`scanner_profiles/bt885.py`; this doc captures the SD-card facts in
one place so future profile work has a clean baseline.

## TL;DR

1. The BT885 is a member of the **`BCDx36HP`** firmware family. Its
   on-disk format is a strict subset of the SDS100/SDS200 layout.
2. **Identity** is `scanner.inf` `Scanner` field 1 = `BT885-SCN`. The
   `TargetModel` header on this scanner is `BCDx36HP`, **not**
   `Beartracker885` (older RE notes / test fixtures that use
   `Beartracker885` were never validated against real hardware).
3. **HPDs** live under `BCDx36HP\HPDB\`. Payload schema is the same as
   the rest of the family; the BT885 simply writes fewer trailing
   feature slots and a `DateModified` header that the SDS100 omits.
4. **Firmware data tables** (`CityTable_V1_00_00.dat`,
   `ZipTable_V1_00_00.dat`) are bit-identical to the SDS100's. They
   are the same database file, parsed by the same code, and could be
   bundled once for the whole family.
5. The BT885's SD card ships with the **Uniden BT885 Update Manager**
   installer at `E:\Setup\` plus its bootstrap dependencies (.NET 4.0,
   Windows Installer 3.1).
6. The BT885 has **no** `profile.cfg`, **no** `app_data.cfg`,
   **no** `discvery.cfg`, **no populated favorites**, and **no**
   QuickKey hardware. The folders for those features
   (`favorites_lists/`, `discovery/`, `audio/`, `activity_log/`,
   `alert/`) exist on the card as empty stubs because the firmware
   shares the BCDx36HP scaffolding.

## `BCDx36HP\scanner.inf`

```
TargetModel	BCDx36HP
FormatVersion	1.00
Scanner	BT885-SCN	<SERIAL>	1.01.02 	01		1.00.00	1.00.00	0
```

| Field after `Scanner` | Value | Meaning |
| ---: | --- | --- |
| 1 | `BT885-SCN` | **Model fingerprint** |
| 2 | `<SERIAL>` | Serial / part number |
| 3 | `1.01.02 ` | MAIN firmware version (trailing space is real) |
| 4 | `01` | Hardware revision |
| 5 | (empty) | Reserved |
| 6 | `1.00.00` | DSP firmware? (matches SDS100 default) |
| 7 | `1.00.00` | DSP firmware? |
| 8 | `0` | Reserved |

The SDS100 adds a 9th field for the SUB-processor firmware. BT885 has
no SUB MCU and the field is absent. Detection / version logic must
treat field 9+ as optional.

Identical shape to the SDS100's `hpdb.cfg`; the BT885's is much

These are all parsed by `scanner_manager.py` `HpdFile._parse`. No

SDS100 sometimes emits do not appear on this BT885 card).

### Field-width comparison vs SDS100

| Record | BT885 fields | SDS100 fields | Notes |
| --- | ---: | ---: | --- |
| `Conventional` | 7 | 14 | SDS100 carries 7 extra empty trailing slots. |
| `C-Group` | 8 | 10 | SDS100 adds `Off / Global` (favorite-list flag, scope). |
| `C-Freq` | 7 | 17 | SDS100 carries lock-out / record-hold / per-channel feature slots. |
| `Site` | ~10 | 13 | SDS100 adds Standard / Wide modulation flags inside favorites. |

The HPD parser preserves all trailing fields verbatim
(`record.fields = parts[1:]`), so round-tripping a BT885 HPD through
the parser is lossless. **Do not "normalize" empty trailing fields**
- the firmware will reject a file whose record widths shift.

## `BCDx36HP\firmware\CityTable_V1_00_00.dat`

566,492 bytes. **Bit-identical** to the SDS100's CityTable on
`H:\BCDx36HP\firmware\CityTable_V1_00_00.dat`
(SHA-256 `e59c0e036c5831b6b16ad909aab50be91dbf837e78cff04e14f6d6a6da1757d3`).

### Layout

```
header bytes ...
START_CITY_TABLE\x00
<payload: fixed-width records>
END_CITY_TABLE\x00
trailer bytes ...
```

The header / trailer carry firmware-table magic and version stamps
that we do not interpret. We only operate on the payload between
the two NUL-terminated ASCII markers.

### Record schema (`FirmwareCityTable` in `scanner_manager.py`)

12 bytes per record (the BT885 image confirms 12 B; the parser
auto-detects 16/20/24 if a future firmware emits padded records):

| Offset | Size | Type | Meaning |
| ---: | ---: | --- | --- |
| 0 | 2 | `[A-Z]{2}` | State / territory abbreviation (e.g. `FL`) |
| 2 | 2 | `>H` (big-endian uint16) | Internal city ID |
| 4 | 4 | `>I` (big-endian uint32) | Latitude raw |
| 8 | 4 | `>I` (big-endian uint32) | Longitude raw |

Coordinate decode:

```
lat = lat_raw / 600000.0 - 90.0
lon = lon_raw / 600000.0 - 360.0
```

Validity check: `-90 <= lat <= 90` and `-180 <= lon <= 180`. Records
with non-ASCII state codes or out-of-range coordinates are skipped.

### Real-card stats

- Records: **47,204**
- Distinct states / territories: **64** (50 US states + DC + territories
  + a few Canadian provinces).
- Detected record size: **12** bytes.

### Re-write rules

`FirmwareCityTable.export_patched()` round-trips the original file
header / trailer verbatim and appends user-supplied `CityRecord`s to
the body. New records are encoded with the same big-endian schema.
**Tail-padding** is auto-added so each new record matches the
detected record size; if the firmware reads padded fixed-width
records, our extras align correctly.

The CityTable is what the BT885's "city" feature uses to label
location-tagged events. We don't currently surface in-app editing for
BT885; the SDS100 RE doc covers a planned UI.

## `BCDx36HP\firmware\ZipTable_V1_00_00.dat`

693,758 bytes. **Bit-identical** to the SDS100's ZipTable
(SHA-256 `69ea964e13eac869e49c7d4043ef94c72092c208d9d79b78d033b6439da41cc9`).

### Layout

```
header bytes ...
START_ZIP_TABLE\x00
<payload: fixed-width records>
END_ZIP_TABLE\x00
trailer bytes ...
```

### Record schema (`FirmwareZipTable` in `scanner_manager.py`)

16 bytes per record on this card; parser autodetects 12/16/20/24.

| Offset | Size | Type | Meaning |
| ---: | ---: | --- | --- |
| 0 | 7 | ASCII (`[A-Z]{2}\d{5}`) | State + 5-digit ZIP, joined |
| 7 | 1 | `uint8` | Flag byte (NUL-terminator on this firmware; reserved for region/sub-division flags on others) |
| 8 | 4 | `>I` | Latitude raw |
| 12 | 4 | `>I` | Longitude raw |
| 16+ | n | bytes | Optional trailing extras (preserved verbatim, not parsed) |

Coordinate decode is the same as CityTable
(`raw/600000 - 90` for lat, `raw/600000 - 360` for lon).

### Real-card stats

- Records: **41,771**
- Coordinate-bearing records: **41,771** (100%)
- Detected record size: **16** bytes
- Flag-byte distribution: `{0: 41,771}` (all records have NUL flag)
- Trailing extras: 0 records have non-zero tail bytes

### Sample records

```
00501  NY  (40.8152, -73.0455)   # Holtsville, NY (IRS distribution center)
00544  NY  (40.8152, -73.0455)
00601  PR  (18.1788, -66.7516)
00602  PR  ...
00603  PR  ...
```

### Use in this app

`scanner_manager.py:FirmwareZipTable` exposes:

- `state_abbrev_for_zip(zip_code) -> Optional[str]`
- `coords_for_zip(zip_code) -> Optional[Tuple[float, float]]`

Both are used by the ZIP/GPS Simulation feature
(`wiki/ZIP-and-GPS-Simulation.md`) to translate a user-entered ZIP
into a center point and state mask. Because the table is shared
across the family, the same simulator works on SDS100 cards.

## What we know is **not** RE'd yet

Items still opaque on the BT885 side:

- The pre-`START_*_TABLE` and post-`END_*_TABLE` bytes in the firmware
  data files (header / trailer). Not parsed; preserved verbatim on
  re-write. Likely a Uniden firmware-table envelope (version stamp +
  CRC?). Worth a hex-dump diff against the SDS100 copy of the same
  file - we already know the **payload** is bit-identical, so any
  envelope difference would be revealed by comparing the leading /
  trailing bytes only.
- The `_extras` byte tail beyond record offset 16 in the ZipTable
  (none observed on this card; the parser keeps any non-zero tail).
- HPD records the parser does not interpret today: `BandPlan_Mot`,
  `FleetMap`. Neither appears in BT885 state HPDs sampled so far, but
  the parser stores them as opaque rows, so RR import that emits
  them will round-trip cleanly.
